> 当前阶段：**Specify（规格定义）** — 仅文档，不含业务代码。

## 1. 项目目标（Objective）

### 1.1 我们在构建什么

一个面向 Web 的**活动报名系统**，支持管理员发布与管理活动，普通用户浏览活动并完成报名、取消报名等操作。

### 1.2 为什么构建

- 为活动组织方提供统一的活动发布、报名审核与名单管理能力
- 为参与者提供便捷的活动浏览与报名渠道
- 通过明确的业务规则与状态机，保证报名流程可控、可追溯

### 1.3 目标用户

| 角色 | 说明 |
|------|------|
| 普通用户 | 浏览活动、报名、查看/取消自己的报名 |
| 管理员 | 活动全生命周期管理、报名名单查看与审核 |

### 1.4 用户故事

**普通用户**

- 作为普通用户，我可以查看已上架的活动列表与详情，以便决定是否参加
- 作为普通用户，我可以对符合条件的活动提交报名，以便参与活动
- 作为普通用户，我可以查看我的报名记录及状态，以便了解审核进度
- 作为普通用户，我可以取消自己的报名，以便在无法参加时释放名额

**管理员**

- 作为管理员，我可以发布、编辑活动，以便维护活动信息
- 作为管理员，我可以上架/下架活动，以便控制活动是否对外可见与可报名
- 作为管理员，我可以查看某活动的报名名单，以便掌握报名情况
- 作为管理员，我可以审核报名（通过/拒绝），以便控制参与人员

## 3. 系统角色与权限

### 3.1 普通用户

| 能力 | 说明 |
|------|------|
| 查看活动 | 浏览已上架活动列表与详情 |
| 报名活动 | 对符合规则的活动提交报名 |
| 查看我的报名 | 查看本人全部报名记录及状态 |
| 取消报名 | 取消本人符合条件的报名 |

### 3.2 管理员

| 能力 | 说明 |
|------|------|
| 发布活动 | 创建新活动 |
| 编辑活动 | 修改活动信息（受业务规则约束） |
| 上下架活动 | 控制活动 `已上架` / `已下架` 状态 |
| 查看报名名单 | 按活动查看所有报名记录 |
| 审核报名 | 将报名状态变更为「已通过」或「已拒绝」 |

### 3.3 权限原则

- 普通用户仅能操作自己的报名数据
- 管理员接口需鉴权，非管理员不可访问
- 已下架活动对普通用户不可见（或不可报名，见开放问题）

## 4. 核心业务规则

以下规则为**硬性约束**，后端必须校验，前端应做友好提示。

| # | 规则 | 说明 |
|---|------|------|
| R1 | 不可重复报名 | 同一用户对同一活动只能存在一条有效报名（待审核或已通过）；已取消/已拒绝后是否可再次报名见开放问题 |
| R2 | 截止后不可报名 | 当前时间超过活动报名截止时间后，禁止新报名 |
| R3 | 人满不可报名 | 活动已通过审核的报名人数达到上限后，禁止新报名 |
| R4 | 下架不可报名 | 活动处于「已下架」状态时，禁止新报名 |
| R5 | 报名状态机 | 报名状态：`待审核`、`已通过`、`已拒绝`、`已取消` |
| R6 | 管理员可审核 | 管理员可将「待审核」报名审核为「已通过」或「已拒绝」 |
| R7 | 统一 JSON 响应 | 所有 API 返回统一 JSON 结构（见第 8 节） |

### 4.1 活动状态（建议）

| 状态 | 说明 |
|------|------|
| 草稿 | 仅管理员可见，未对外发布 |
| 已上架 | 普通用户可见，可报名（若满足其他规则） |
| 已下架 | 不可报名；是否对普通用户可见见开放问题 |

### 4.2 报名状态流转

```
                    ┌─────────────┐
                    │   待审核    │
                    └──────┬──────┘
           ┌───────────────┼───────────────┐
           ▼               ▼               ▼
    ┌─────────────┐ ┌─────────────┐ ┌─────────────┐
    │   已通过    │ │   已拒绝    │ │   已取消    │
    └─────────────┘ └─────────────┘ └─────────────┘
         ▲                               ▲
         │         用户取消（待审核/已通过）│
         └───────────────────────────────┘
```

**流转说明（建议）**

| 当前状态 | 动作 | 目标状态 | 执行者 |
|----------|------|----------|--------|
| — | 提交报名 | 待审核 | 普通用户 |
| 待审核 | 审核通过 | 已通过 | 管理员 |
| 待审核 | 审核拒绝 | 已拒绝 | 管理员 |
| 待审核 | 取消报名 | 已取消 | 普通用户 |
| 已通过 | 取消报名 | 已取消 | 普通用户 |
| 已拒绝 | — | 终态 | — |
| 已取消 | — | 终态 | — |

## 5. 领域模型（概念）

实现前的概念模型，具体字段在详细规格阶段细化。

### 5.1 活动（Activity）

- 标题、描述、地点/线上说明
- 报名开始时间、报名截止时间
- 活动开始时间、活动结束时间（可选）
- 人数上限（`maxParticipants`）
- 当前已通过人数（`approvedCount`，可冗余计数或实时统计）
- 状态：草稿 / 已上架 / 已下架
- 创建人（管理员）

### 5.2 用户（User）

- 基本信息：用户名、密码（加密存储）、昵称等
- 角色：普通用户 / 管理员

### 5.3 报名（Registration）

- 关联用户、关联活动
- 状态：待审核 / 已通过 / 已拒绝 / 已取消
- 报名时间、审核时间、审核人（管理员）
- 唯一约束：`(user_id, activity_id)` 在有效报名语义下的处理策略见开放问题

## 8. API 约定

### 8.1 统一响应格式

所有接口返回 JSON，建议结构：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `code` | number | `0` 表示成功，非 `0` 为业务或系统错误码 |
| `message` | string | 人类可读的提示信息 |
| `data` | object / array / null | 业务数据；列表接口可包含分页字段 |

**错误示例**

```json
{
  "code": 40001,
  "message": "活动报名已截止",
  "data": null
}
```

### 8.2 REST 风格（概要，详细 API 规格另文档）

| 模块 | 方法 | 路径（示例） | 角色 |
|------|------|--------------|------|
| 活动列表 | GET | `/api/activities` | 用户 |
| 活动详情 | GET | `/api/activities/{id}` | 用户 |
| 发布活动 | POST | `/api/admin/activities` | 管理员 |
| 编辑活动 | PUT | `/api/admin/activities/{id}` | 管理员 |
| 上下架 | PATCH | `/api/admin/activities/{id}/status` | 管理员 |
| 报名 | POST | `/api/registrations` | 用户 |
| 我的报名 | GET | `/api/registrations/mine` | 用户 |
| 取消报名 | PATCH | `/api/registrations/{id}/cancel` | 用户 |
| 报名名单 | GET | `/api/admin/activities/{id}/registrations` | 管理员 |
| 审核报名 | PATCH | `/api/admin/registrations/{id}/audit` | 管理员 |

## 9. 代码风格（Code Style）

### 9.1 后端（Java / Spring Boot）

- 包名小写，类名大驼峰：`ActivityController`、`RegistrationService`
- REST 控制器薄层，业务逻辑放在 Service
- 业务规则校验失败抛出自定义业务异常，由全局异常处理器转换为统一 JSON
- 数据库表名、字段名：`snake_case`；Java 实体：`camelCase`

**示例：业务异常与统一响应**

```java
// 业务规则：报名截止后不可报名
if (LocalDateTime.now().isAfter(activity.getRegistrationDeadline())) {
    throw new BusinessException(40002, "活动报名已截止");
}
return ApiResponse.success(registration);
```

### 9.2 前端（Vue 3 + TypeScript）

- 使用 Composition API + `<script setup lang="ts">`
- 组件文件 PascalCase：`ActivityCard.vue`
- API 调用集中在 `src/api/`，不在组件内散落 `fetch`
- 与后端对齐的类型定义放在 `src/types/`

**示例：调用统一响应**

```typescript
interface ApiResponse<T> {
  code: number
  message: string
  data: T
}

async function registerActivity(activityId: number) {
  const res = await api.post<ApiResponse<Registration>>('/registrations', { activityId })
  if (res.code !== 0) {
    throw new Error(res.message)
  }
  return res.data
}
```

## 11. 边界与协作约定（Boundaries）

### 11.1 Always（始终遵守）

- 所有接口返回统一 JSON 结构（R7）
- 后端对 R1–R6 做服务端校验，不依赖前端
- 提交代码前运行相关测试与 lint
- 遵循本文档与后续 `docs/` 下规格文档的约定
- 敏感配置（数据库密码等）不入库，使用环境变量或本地配置

### 11.2 Ask First（先确认再改）

- 数据库表结构变更、迁移脚本
- 新增主要依赖或更换技术选型（如 ORM、认证方案）
- 修改统一 API 响应结构或错误码体系
- 变更核心业务规则或状态机
- CI/CD 配置变更

### 11.3 Never（禁止）

- 在仓库中提交密钥、生产环境密码
- 为通过测试而删除或跳过失败用例（未经确认）
- 绕过权限校验暴露管理员接口
- 在未更新文档的情况下擅自扩展 scope

## 12. 成功标准（Success Criteria）

系统达到可交付时，应满足：

- [ ] 普通用户可浏览已上架活动、报名、查看我的报名、取消报名
- [ ] 管理员可发布/编辑活动、上下架、查看报名名单、审核报名
- [ ] R1–R7 全部业务规则在后端 enforced，并有测试覆盖
- [ ] 前后端所有接口符合统一 JSON 响应约定
- [ ] 角色权限隔离：普通用户无法访问管理端 API
- [ ] 主要页面在主流浏览器可正常使用（Chrome、Edge、Firefox 近期版本）

## 13. 假设与开放问题（Assumptions & Open Questions）

### 13.1 当前假设（请 Review 后确认或纠正）

1. **Web 应用**：仅面向浏览器，不含原生 App
2. **用户体系**：需要注册/登录；管理员账号由系统预置或超级管理员创建，不支持自助提权
3. **认证方式**：暂定为 Session Cookie 或 JWT 二选一，实现前确认
4. **「有效报名」定义**：同一用户对同一活动，若存在「待审核」或「已通过」记录，则视为不可重复报名；「已拒绝」「已取消」是否允许再次报名——**默认允许再次报名**，若不允许需明确
5. **人数统计**：`已通过` 状态计入人数上限；`待审核` 是否占用名额——**默认不占用**，审核通过时再校验名额
6. **下架活动展示**：普通用户列表不展示已下架活动；直接访问详情返回 404 或「活动不存在」
7. **取消报名**：`待审核` 与 `已通过` 均可取消；`已通过` 取消后释放名额
8. **时区**：系统使用服务器时区或统一 UTC 存储、前端本地化展示——实现前确认
9. **项目为新建**：当前仓库尚无 backend/frontend 代码，结构按第 6 节规划

### 13.2 待确认问题

| # | 问题 | 影响 |
|---|------|------|
| Q1 | 认证用 Session 还是 JWT？ | 前后端联调、部署方式 |
| Q2 | 已拒绝/已取消后是否允许同一用户再次报名同一活动？ | R1 的数据库唯一约束设计 |
| Q3 | 待审核报名是否占用活动名额？ | 并发报名与超员逻辑 |
| Q4 | 活动是否需要「报名开始时间」？还是仅「截止时间」？ | 活动模型字段 |
| Q5 | 是否需要报名备注/自定义表单字段？ | 报名表单复杂度 |
| Q6 | 是否需要邮件/站内通知（审核结果）？ | 通知模块 scope |
